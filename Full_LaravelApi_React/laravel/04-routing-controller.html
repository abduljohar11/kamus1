Validasi Layer API

Mengamankan Input Data dengan Form Request

Sesi 2: Membuat aturan validasi yang ketat untuk aksi registrasi (Store) dan modifikasi (Update) user, termasuk penanganan validasi unik pada tipe Primary Key ULID.

LOGIKA FILTERING INPUT

Ketika HTTP Request masuk membawa payload JSON, Laravel akan mencocokkannya dengan rules di dalam Form Request. Jika validasi gagal, Laravel otomatis membatalkan eksekusi ke Controller dan langsung melempar HTTP Response 422 Unprocessable Entity beserta daftar error-nya dalam format JSON yang bersih.

1 StoreUserRequest (Aksi Create)

Gunakan perintah php artisan make:request StoreUserRequest untuk membuat berkas ini di dalam direktori app/Http/Requests/.

Sintaks Penuh: app/Http/Requests/StoreUserRequest.php 
<?php

namespace App\Http\Requests;

use Illuminate\Foundation\Http\FormRequest;
use Illuminate\Validation\Rule;

class StoreUserRequest extends FormRequest
{
    /**
     * Tentukan apakah user diizinkan melakukan request ini.
     * Logika: Ubah menjadi 'true' jika tidak menggunakan kebijakan Gate/Policy di layer ini.
     */
    public function authorize(): bool
    {
        return true;
    }

    /**
     * Aturan validasi yang diterapkan pada payload request.
     */
    public function rules(): array
    {
        return [
            'name'        => ['required', 'string', 'max:255'],
            'email'       => ['required', 'string', 'email', 'max:255', 'unique:users,email'],
            'password'    => ['required', 'string', 'min:8'],
            'role'        => ['required', 'string', Rule::in(['guru', 'murid'])],
            'nomor_induk' => ['required', 'string', 'max:50', 'unique:users,nomor_induk'],
        ];
    }
}

2 UpdateUserRequest (Aksi Update)

Logika pengecualian (*ignore rule*) sangat penting saat update. Jika user tidak mengubah email atau nomor induk mereka, sistem tidak boleh memicu error "sudah terpakai". Buat file ini di app/Http/Requests/UpdateUserRequest.php.

Sintaks Penuh: app/Http/Requests/UpdateUserRequest.php 
<?php

namespace App\Http\Requests;

use Illuminate\Foundation\Http\FormRequest;
use Illuminate\Validation\Rule;

class UpdateUserRequest extends FormRequest
{
    public function authorize(): bool
    {
        return true;
    }

    public function rules(): array
    {
        // Menangkap parameter id (ULID) user dari segmen route URL yang sedang diakses
        $userId = $this->route('user');

        return [
            'name'        => ['sometimes', 'required', 'string', 'max:255'],
            // Logika: Abaikan pengecekan unik jika email milik id user itu sendiri
            'email'       => ['sometimes', 'required', 'string', 'email', 'max:255', Rule::unique('users', 'email')->ignore($userId)],
            'password'    => ['nullable', 'string', 'min:8'],
            'role'        => ['sometimes', 'required', 'string', Rule::in(['guru', 'murid'])],
            'nomor_induk' => ['sometimes', 'required', 'string', 'max:50', Rule::unique('users', 'nomor_induk')->ignore($userId)],
        ];
    }
}

Analisis Mekanisme Validasi

'sometimes' Rule

Aturan ini memberi tahu Laravel untuk menjalankan validasi hanya jika *key* tersebut dikirim di dalam payload data. Sangat ideal untuk request bertipe PATCH/PUT yang seringkali hanya mengirimkan sebagian kolom data profile.

->ignore($userId)

Tanpa adanya method ini, kueri database unik milik Laravel akan menolak proses update karena email atau NIP/NISN yang bersangkutan sudah tercatat di baris data Anda sendiri sebelumnya.

'password' => 'nullable'

Memungkinkan frontend untuk mengirimkan string kosong pada input password jika user hanya ingin mengganti nama atau email saja tanpa berniat memodifikasi kredensial login mereka.
